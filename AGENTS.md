# AGENTS.md — スピリチュアル系リサーチ（本ディレクトリ）

人間向けの説明・推奨フローは [[00_はじめに]]。本ファイルは **エージェントが迷わず編集・追記するための契約** を書く。

## 目的（1行）

YouTube スピ系（金運・潜在意識等）の **市場リサーチ** を、`伸びる型の抽出` と `自チャンネル設計への変換` まで含めて行い、ジャンル単位の結論（`_genre_sheet.md`）に集約する。

## このディレクトリで達成すること

- 動画1本の感想で終わらせず、`クリックの約束 / 信頼形成 / 参加・儀式 / 再視聴導線 / 収益導線` に分解して、**なぜ伸びるか** を言語化する。
- 属人依存の要素と、テンプレ化して移植できる要素を分け、**自分のチャンネルの勝ちパターン** に変換する。
- 単発の気づきは `research.md` や `調査_CH###` に残し、**結論は `_genre_sheet.md` に集約** して次の企画判断に使える形にする。

## ユーザー誘導の原則

- 要約依頼でも、可能なら **「なぜこの構成なのか」** を説明する。
- ユーザーが学習目的を持っている場合は、各主要フェーズで短い **アウトプット用の問い** を入れて、自分の仮説を書けるようにする。
- 1本の動画を扱ったら、最後は `そのまま使う / 変えて使う / 捨てる` の3つに整理して、自チャンネルへの示唆まで出す。
- 迷ったら、単なる感想の網羅よりも **再現可能な勝ち筋の抽象化** を優先する。

## セッション開始（必須）：Git で最新を取り込む

**どの運用・作業の前でも、まずここから行う。** 本フォルダが Git リポジトリのルート（例: `spiritualResearch`）のときは、そのディレクトリで実行する。

```powershell
git fetch
git pull
```

- **目的**: 他端末・他者・別ブランチの更新を取り込み、`30_チャンネル調査/` や `_registry.md` の採番と衝突しないようにする。
- **pull でコンフリクトしたら**: 自動解決せず、人間に報告して指示を待つ。
- **親 Vault からだけ触る場合**も、リポジトリルートを開いて同様に `fetch` / `pull` してから編集する。

## 読む順序（`git fetch` / `git pull` の直後）

1. 本 `AGENTS.md`（制約）
2. [[00_はじめに]]（人間向けフローとフォルダ早見表）
3. 変更対象に応じて [[_registry]] または [[_genre_registry]]

## 動画分析をするとき（チャンネルが一覧にない場合）

**チャンネル一覧** ＝ [[_registry]] の表 ＋ `30_チャンネル調査/CH###_…/` フォルダの有無で判断する。

対象動画のチャンネルが **未登録**（`CH###` がない／台帳にチャンネル名がない）のときは、**分析用の `research.md` を書く前に** 次を済ませる。

1. 既存の最大番号を確認し、次の `CH###` を採番する。
2. `30_チャンネル調査/CH###_チャンネル名/` を作成する。
3. `10_テンプレート/チャンネル_index_テンプレート.md` を元に `_channel_index.md` を置き、URL・主ジャンル（`G###`）などを埋める。
4. [[_registry]] に追記する（まだ動画フォルダだけでない場合は「チャンネルのみ」小表、または最初の `VID###` まで決まっていれば本表に `CH###-VID###` 行）。
5. そのうえで `VID###_…/research.md` を作成して動画分析を行う。

すでに `CH###` があるチャンネルの **新しい動画** だけの場合は、`_registry.md` に `VID###` 行を足し、既存 `CH###` 配下に `VID###` フォルダを追加する（採番ルールは「よくある作業」を参照）。

## 作業後の Git（推奨）

ユーザーの運用に任せるが、変更を共有するならリポジトリルートで `git add` → `git commit` → `git push`。エージェントは **コミットメッセージに何を変えたか** を一文残せるとよい。

## 単一の真実（SoT）

| 種類 | SoT | エージェントの扱い |
|---|---|---|
| ジャンル市場の結論 | `20_ジャンル/G###/…/_genre_sheet.md` | 分析の最終反映先。チャンネル分析後は §4〜6・§5 を更新する |
| チャンネル一覧（動画あり） | ルートの `_registry.md` | 新規 `VID` のたびに行追加。パスは `30_チャンネル調査/…` 基準 |
| チャンネルのみ先登録 | `_registry.md` 内「チャンネルのみ登録」小表 | 動画 `research.md` ができたら本表へ移し、小表から削除 |
| ジャンル ID | `_genre_registry.md` + `20_ジャンル/_genre_index.md` | 新規 `G###` は両方更新 |

## 補助メモと重複防止

| 種類 | 置き場 | ルール |
|---|---|---|
| 市場のたたき台・散文メモ | `20_ジャンル/G###/メモ_市場スケッチ_G###.md`（任意） | **結論は `_genre_sheet.md` にしか書かない**。メモから本番へ移したら、表・パターンは §3〜6 に集約する |
| 動画の本分析 | `30_チャンネル調査/…/VID###/research.md` | ジャンルに載せる行は `_genre_sheet` か `調査_CH###` 経由 |
| 動画の概要欄全文 | 同フォルダの `description.md`（任意だが推奨） | 手入力 or YouTube Data API `videos.list` の `snippet.description`。分析は `research.md` §3「概要欄分析」 |
| 旧フォルダ名 | `01_ジャンルマップ/` | **使用禁止・再作成しない**（`20_ジャンル` に統合済み） |

同じ動画について `メモ_*`・`research.md`・`_genre_sheet` に重複記述が出たら、**表とチャンネル表は `_genre_sheet` を正**とする。

## ID 規約（変更禁止の意味）

- `G###` … ジャンル固定。フォルダ名は `20_ジャンル/G###_スラッグ/`。
- `CH###` … チャンネル固定。フォルダは `30_チャンネル調査/CH###_チャンネル名/`。
- `CH###-VID###` … チャンネル内の動画連番。
- `TH-CH###-VID###-01` … サムネ ID。画像は当該動画フォルダの `assets/`。

採番は既存台帳の **最大番号+1** で揃える。

## エージェント向け禁止事項

- ジャンル結論を **チャンネルフォルダだけ** に置いて `_genre_sheet.md` を空のままにしない（結論はジャンルシートへ）。
- `_registry.md` のパスをルート直下の `CH###` に戻さない（必ず `30_チャンネル調査/` 配下）。
- 同一ルールを Vault 外の別ファイルに長文コピーしない（SoT は本ディレクトリ内）。
- `01_ジャンルマップ/` を復活させない（空の `_genre_sheet` が二重管理の原因になる）。

## 人間への引き渡し

作業終了時は、変更した `G###` / `CH###` と、次に人が確認すべき一次情報（例: YouTube での登録者数の再確認）があれば短くメモする。
