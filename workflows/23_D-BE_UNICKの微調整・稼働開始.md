## このレポートで分かること

Phase 3 の**締め**。パーツを LINE 周りに接続し、**テスト配信 → 本番 → 初期チューニング**までの考え方とチェックリスト（教材）。

## 前提：パーツが揃っている状態（教材）

キャラ・商品・戦略シート・ステップ原稿（例：**10通**／設計により前後）・LINE 登録特典・LP。あとは **LINE 公式**と**自動化ツール**（必要なら）に載せ、**動作確認してから**本番公開する。

## よくある勘違い（教材）

**接続した＝完成**としてテスト・改善を省略すると、リンク不備や表示崩れで機会損失が出やすい。**稼働後も**数値を見ながら小さく改善する前提が安全。

---

## なぜテスト・チューニングか（教材）

1. **リンク不備は売上に直撃** — 決済・特典 URL は**実タップで**確認。  
2. **スマホ LINE が正** — 改行・見た目は**実機**でしか分からない部分がある。  
3. **初稿は仮説** — 母数を取ってから**要素ごと**に手を入れる。

## システムへの実装（教材）

LINE 公式単体でもステップは可能だが、**タグ・分岐・計測**を厚くするなら拡張ツール（**Lステップ、エルメ、マイスピー、L-Message** 等）の利用が選択肢。**機能名・仕様は各サービスが更新**するため、導入前に公式で確認。

| 観点 | 例 |
|------|-----|
| タグ | 配信到達・リンクタップ・離脱目安の記録 |
| 分岐 | アンケート・行動に応じたルート |
| 分析 | 通別の開封・クリックの把握（取れる指標はツール依存） |

### 実装の流れ（5ステップ・教材）

1. ツール登録と LINE 公式との連携（**正本は `05`**）  
2. **シナリオの枠** — 「○日目・○時」など**設計どおり**の本数・間隔（**開封が取りやすい時間帯は検証で決める**）  
3. **原稿の流し込み** — [`20_配列設計と本文生成.md`](20_配列設計と本文生成.md) で整えた本文を反映  
4. **特典リンク** — 第1話（または特典案内通）で**届く・開ける**ことを確認  
5. **商品リンク** — オファー箇所（例：**第8〜10話**／**採用配列は `19`・`05` に合わせる**）に決済・販売ページ URL を配置

## テスト配信の考え方（教材）

**自分が読者になりきる**。PC だけでなく**スマホ**（できれば **iPhone と Android**）で確認。

### 1. テスト用アカウント

本番と分けた LINE で受信。可能なら機種を分ける。

### 2. LP → 登録フロー

- 登録導線から友だち追加まで  
- 登録直後の**あいさつ／第1話**  
- **特典 URL** が開き、**スマホで読める**か  

### 3. 全通の通し確認

拡張ツールの**テスト配信**等で、待ち時間を短縮して**順序・改行・リンク**を確認。  
**第8〜10話**（またはオファー通）の**販売リンク**は**実際に開く**こと。可能なら**テスト購入**まで。

### テスト → 本番の流れ（教材）

```text
Phase A：実装（シナリオ・本文・リンク）
  ↓
Phase B：テスト（LP→登録、全通、リンク・決済）
  ↓ NG → 修正して再テスト
Phase C：本番（LP 公開・集客開始、一定母数の収集）
  ↓
Phase D：チューニング（KPI 観察 → 仮説 → 1〜2箇所だけ変更）
```

**成約率を「3〜10%」等と語る例**は**教材の目安**。単価・ジャンル・流入品質で変動する。

## 死活・品質チェックリスト（教材）

### 死活（1つでも NG なら本番見送り推奨）

- [ ] 登録導線から**友だち追加**まで問題ない  
- [ ] 登録直後に**第1話（あいさつ）**と**特典案内**が届く設計どおりか  
- [ ] 特典リンクが**エラーなく**開く  
- [ ] オファー通の**販売リンク**が開く  
- [ ] 決済フローが**テストで**最後まで通る（可能な範囲で）  

### 品質

- [ ] スマホで**文字サイズ**・余白は読めるか  
- [ ] **改行**は崩れていないか  
- [ ] 1通が**長すぎない**か（LINE の仕様・読みやすさの上限を意識）  
- [ ] 絵文字・記号の**表示**は問題ないか  

### 悪いテスト vs 良いテスト（教材）

| 項目 | 悪い方向 | 良い方向 |
|------|----------|----------|
| 環境 | PC のみ | **スマホ実機** |
| 範囲 | 1通だけ | **全通**＋リンク |
| リンク | 見た目だけ | **タップで開く** |
| 決済 | ページ表示のみ | **テスト購入**まで可能なら |
| 確認者 | 自分のみ | 可能なら**第三者** |

## 稼働後のチューニング（8要素の微調整・教材）

データを見て **N・I・E・B・U・K** など**どこが詰まりか**を仮説化し、**1回の改善で 1〜2 箇所**に絞る（要因が分離しやすい）。

### 追う KPI の例（※すべて目安・ツールで定義が異なる）

| 区分 | 指標の例 | 教材での目安（参考） |
|------|----------|----------------------|
| 開封（既読） | 序盤〜中盤〜終盤 | 高いほど良いが、**業種・文言で変動** |
| CTR | 特典／中盤リンク／販売リンク | **母数が少ないと振れ大** |
| 成約率 | 登録対購入 | **単価が高いほど低く出やすい** |

**10人未満で断定**しにくい。**数十人規模**で方向性を見るのが無難（[`30`](30_データ分析と改善サイクル.md) も参照）。

### チューニング例（教材・倫理・事実と整合）

| 兆候 | 仮説の例 | 対応の例 |
|------|----------|----------|
| 第2〜3通付近で離脱が大きい | **N** が強すぎて拒否感／**I** の納得が弱い | N を**共感寄り**に、I の直後に **E（裏づけ）** を足す |
| 最後まで読むが売れない | **B** の魅力不足／**U（必然性）** が弱く先延ばし | B を**具体化**、U は**事実に基づく「なぜ今」**（虚偽の締切は不可） |
| 開封はあるがクリックが少ない | **K** が曖昧 | **何を・どこをタップするか**を一文で明示 |

### 優先度の目安（教材）

1. **今すぐ** — リンク切れ、決済エラー、序盤開封の異常低下  
2. **数日〜1週間** — 序盤離脱の急増、特典取得率の低さ、中盤開封の低さ  
3. **数週間** — 最後まで読むが成約しない → **B・U・K**、または**販売ページ／商品・集客**の切り分け  

### 判断の流れ（教材）

成約が目標に届かないときの**一例**：

1. **開封**がボトルネック → 冒頭・**D**・通知の出方・配信時間の検証  
2. **クリック**がボトルネック → **K**・**B**・リンクの見せ方  
3. **販売ページ離脱**が高い → **販売 LP／チェックアウト**・価格・オファー整合  
4. 上記でも弱い → **商品需要・集客の質**の再検討

## つまずき（稼働時・教材）

1. **テスト省略で公開**  
2. **数値なし**で原稿を毎週全面改稿  
3. **母数不足**で勝ち負けを決める  
4. **一度に全部**書き換え（何が効いたか不明）  
5. **1回の数字**で手を止める（改善サイクルは継続物）

## FAQ（要約）

| Q | A（教材） |
|---|-----------|
| テストは何回？ | **自分の通し**＋**全リンク**＋**本番相当の1通**が最低ライン。余力で第三者。 |

| チューニングして伸びない | **集客の質** → **商品・価格** → **販売ページ** → **原稿の限界**の順で切り分け。 |
