# ターゲットリサーチ実行スキル

## このスキルの目的

どのジャンルでも使える**汎用の悩み大量棚卸しシステム**を使い、ターゲット（見込み顧客・読者・フォロワー）の悩みを100〜150個レベルで一気に列挙し、使える形に整理する。

このスキルは「参照」ではなく「実行」である。
各Phaseを順番に実行し、Phaseを飛ばすことは禁止する。

---

## 絶対原則（実行前に必ず頭に入れる）

```
悩みリサーチの本質は「本人語」で出すこと。

× 抽象語：「将来が不安」「つらい」「モヤモヤ」
○ 本人語：「このまま続けても成果が出る気がせず、時間だけ失って終わりそうで怖い」

悩みの定義：
「本人が日常で口にする不満・不安・迷い・つまずき・恥・恐れ・面倒・焦り」
「何が」「どの場面で」「何に困るか」が分かる文にする。
```

---

## 前提条件

| ファイル | 用途 | 必須度 |
|---------|------|--------|
| キャラクター設計書（完全版） | 専門領域・ターゲット像の参照 | 推奨（なくても実行可） |
| `ターゲット設計テンプレート.md` | 出力フォーマットの参照 | 参照 |

キャラクター設計書がなくても、ジャンル指定だけで実行できる。

---

## 起動条件

ユーザーが以下のいずれかの意図を示したとき：
- 「ターゲットをリサーチして」
- 「ターゲットの悩みを調べて」
- 「悩みを棚卸ししたい」
- 「ペルソナを設計して」
- ジャンルを指定して、読者像・悩みの洗い出しを求めた場合

---

## 実行フロー全体像（全Phase必須・飛ばし禁止）

```
Phase A: 入力情報の準備（初心者は補助フローあり）
    |
Phase B: 悩みの大量発散（100〜150個）← 汎用マスタープロンプト
    |
Phase C: 追加モジュールで増量・補正（必要なものだけ）
    |
Phase D: 整形・クラスター化（使える形にする）
    |
Phase E: Deep Research版で根拠補強（任意）
    |
Phase F: ターゲットペルソナの設計
    |
Phase G: ユーザー確認 + 最終出力
```

## Phase A: 入力情報の準備

### A-1: ユーザーに入力してもらう項目

```
ターゲットリサーチを開始します。以下を埋めてください。

【最低限これだけでOK（3点）】
1. ジャンル：
2. 顧客の望むゴール：
3. 顧客の制約：

【精度を上げたい場合（任意）】
4. 顧客の状態：
5. 想定する代表セグメント：
6. 避けたい表現やNG：
```

### A-2: 全体のコツ（初心者向け）

```
6項目ぜんぶ埋めなくてOK。初心者はこれだけで動く：

- ジャンル：何の相談か
- 顧客の望むゴール：どうなりたいか
- 顧客の制約：何が邪魔してるか（時間・金・自信だけでOK）

それ以外（状態、セグメント、NG）は「不明」でも回せる。
むしろ、不明のまま一回出して、出力を見てから埋め直すのが正攻法。

=== 10分ルール ===
1. 6項目は10分だけ悩む
2. 埋まらないところは「不明」or「仮」で埋める
3. そのまま悩みを100個出す
4. 出てきた悩みを見て「あ、こういう人だな」が分かったら入力欄を修正

入力欄は最初から当てにいくものじゃなく、生成結果で"逆に分かる"もの。
```

### A-3: 各項目が埋められない場合の補助フロー

初心者がテンプレートを埋められない場合、以下のAI補助プロンプトを使って支援する。

#### ジャンルの補助

ジャンルは「テーマ名」じゃなく「誰が／何を／どの方向に」の形にするとブレない。

例：
- 恋愛 → 「恋人を作りたい人の恋愛」
- 副業 → 「会社員が在宅で月5万作る副業」
- 転職 → 「未経験で職種チェンジしたい転職」

埋められない場合のAI補助：

```text
私はジャンルを決めたい初心者です。
「コンサルのジャンル」を、ブレにくい定義（誰が/何を/どの方向に）で
候補を10個出してください。

私が今考えているテーマ：
{{思いつく単語を1〜3個でOK 例：副業、SNS、在宅}}
```

#### 顧客の状態の補助

状態は「ステージ + 経験」の2軸で決め打ちする。

- ステージ：まだ始めてない／始めた／続けてる／伸びない／挫折した
- 経験：初心者／中級／経験者

迷ったら「初期で停滞」に置くと悩みが一番出やすい。

埋められない場合のAI補助：

```text
ジャンルは{{ジャンル}}です。
見込み顧客の「状態」を、ステージ×経験で整理したいです。

ステージ（開始前/情報収集/初期行動/継続/停滞/失敗後/再挑戦/長期運用）
×
経験（初心者/中級/経験者）
で、代表的な状態を12個提案してください。

その中から「悩みが最も多そう」な上位3つも理由つきで選んでください。
```

#### 顧客の望むゴールの補助

ゴールは「成果 + 期限 + 理想」で考える。

例：
- 恋愛：3ヶ月以内に安定交際に入りたい
- 副業：まず月3〜5万を安定させたい
- 営業：受注率を上げて数字のプレッシャーを減らしたい

埋められない場合のAI補助：

```text
ジャンルは{{ジャンル}}。
見込み顧客が望みそうなゴールを「具体的」に20個出してください。

条件：
- 数字/期限/状態のいずれかを入れる
- 初心者が言いそうな口調
- 短期ゴール10個＋中期ゴール10個
```

#### 顧客の制約の補助

制約は「時間・金・スキル・環境・心理」の5つから選ぶだけ。

- 時間：忙しい／継続できない
- お金：投資できない／損したくない
- スキル：何ができるか分からない
- 環境：家族・職場・地域・機材
- 心理：怖い／自信ない／失敗したくない

初心者なら「上位3つ」を仮置きすれば十分。
例：**時間がない／何が正解か分からない／失敗が怖い**

埋められない場合のAI補助：

```text
ジャンルは{{ジャンル}}。
見込み顧客が持ちがちな制約を、
「時間/お金/スキル/環境/心理」の5カテゴリでそれぞれ5個ずつ出してください。

さらに、そのジャンルで特に致命的になりやすい制約Top5を理由つきで選んでください。
```

#### 代表セグメントの補助（任意だが強い）

セグメントは「2軸だけ」にする。欲張ると破綻する。

おすすめの2軸（どのジャンルでも効く）：
- 初心者／経験者
- ひとりで頑張る派／伴走が欲しい派
- 短期で結果派／長期で積み上げ派
- リスク回避派／チャレンジ派

最初は3セグメントでOK（例：初心者×慎重、初心者×行動、中級×停滞）

埋められない場合のAI補助：

```text
ジャンルは{{ジャンル}}です。
見込み顧客のセグメントを「2軸」だけで切りたいです。

汎用性の高い2軸を3案提案し、
それぞれで4象限（4タイプ）の特徴と典型的な悩みを5つずつ出してください。
```

#### 避けたい表現やNGの補助（任意だが事故防止）

初心者なら最初はこれだけで十分：
- 人格否定しない
- 断定しない（絶対/必ずを避ける）
- 煽りすぎない（恐怖マーケ全振りを避ける）

埋められない場合のAI補助：

```text
ジャンルは{{ジャンル}}。
見込み顧客に対して「避けたい表現・NG」を作りたいです。

- トーン面のNG（言い方）
- 倫理/誇大/断定のNG（内容）
に分けて、それぞれ10個ずつ提案してください。

さらに、代替の言い換え例も付けてください。
```

### A-4: それでも埋められない場合 → AIに穴埋めさせる

6項目を悩んで止まるくらいなら、AIにこう頼むのが一番早い。

```text
私は初心者です。以下の情報だけで、
テンプレ入力（ジャンル/顧客の状態/ゴール/制約/代表セグメント/NG）を
「仮置きで」3パターン作ってください。

私が分かっていること：
- ジャンルの候補：{{例：副業、恋愛、転職のどれか}}
- こういう人を助けたい：{{一言でOK}}
- 絶対にやりたくないこと：{{あれば}}

条件：
- 仮置きでよいが、なぜそう置いたか理由も一言添える
- 3パターンはターゲットの切り口を変える
```

→ 出てきた3パターンのうち「しっくりくる」やつを選んで、Phase Bへ。

## Phase B: 悩みの大量発散（100〜150個）（飛ばし禁止）

### このPhaseの目的

入力情報を元に、ターゲットの悩みを**最低100個、可能なら150個**、本人語で一気に列挙する。

### 汎用マスタープロンプト（原文のまま実行）

Phase Aで埋めた入力情報を `{{ }}` に挿入して、以下をそのまま実行する。

```text
あなたは「顧客の悩みを大量に棚卸しするブレスト担当」です。
目的は、{{ジャンル}}コンサルの見込み顧客が抱える悩みを、具体的な本人語で大量に列挙し、カテゴリ別に整理することです。

【対象（ターゲット）】
- 相談領域（ジャンル）：{{例：恋愛 / 副業 / 転職 / 資格 / 起業 / 営業 / ダイエット など}}
- 顧客の状態（ざっくりでOK）：{{例：初心者〜中級 / 伸び悩み / 失敗経験あり / 自信がない など}}
- 顧客の望むゴール：{{例：交際したい / 月5万稼ぐ / 3ヶ月で転職内定 など}}
- 顧客の制約：{{例：時間がない / お金がない / 家族が反対 / スキル不足 など}}
※不明なら推測してよいが、その場合は「推測」と明記する。

【悩みの定義】
「本人が日常で口にする不満・不安・迷い・つまずき・恥・恐れ・面倒・焦り」を悩みとする。
抽象語だけで終わらせず、「何が」「どの場面で」「何に困るか」が分かる文にする。

【生成ルール】
- 1行＝悩み1つ（本人の一人称）
- "抽象語"禁止（例：不安、つらい、モヤモヤ → 何が不安かまで書く）
- 重複OK。ただし「状況・言い回し」を変えて粒度を増やす
- 最低100個。可能なら150個
- 以下の軸を必ず満遍なくカバーする（偏り防止）：
  1) ステージ：開始前/情報収集/初期行動/継続中/停滞/失敗後/再挑戦/長期運用
  2) 種類：行動（何をすれば）/判断（どれが正解）/感情（怖さ焦り）/対人（相手・周囲）/環境（時間・金）/能力（スキル不足）/リスク（失敗・損）
  3) セグメント：初心者/経験者、短期志向/長期志向、慎重派/行動派、自己流派/指導希望、コスト敏感/投資できる

【出力形式】
(1) まずカテゴリ一覧を提示（あなたが作る）
(2) 次にカテゴリごとに箇条書きで悩みを列挙
(3) 最後に「特に刺さりが強い悩み Top20」を選び、各悩みに1行で典型シーンも付ける

【カテゴリ例（必要なら改変OK）】
開始前の心理 / 情報収集・比較 / 初期のつまずき / 継続・習慣化 / 停滞・伸び悩み / 失敗・挫折 / 対人・コミュニケーション / お金・時間・環境 / 自信・自己肯定感 / 将来・長期不安
```

### 出力が少なかった場合の追い打ち

```
150個まで増やして
```

### 成功している出力の目安

```
チェックリスト：
□ 悩みが「不安です」みたいな抽象で終わっていないか
□ 「いつ・どこで・何が起きて困る」が想像できるか
□ 特定のステージの悩みばかりに偏っていないか
□ Top20が「確かにこれ刺さる」と思えるか
```

## Phase C: 追加モジュールで増量・補正（必要なものだけ）

Phase Bの出力が「弱い」と感じたら、ここで補強する。
全部やる必要はない。困っている症状で選ぶ。

### 症状 → 選ぶモジュール 対応表

| 症状 | 使うモジュール |
|------|--------------|
| 抽象的すぎる | A. 抽象→具体矯正モジュール |
| 本人の言い方っぽくない | B. 独り言増量モジュール |
| 表に出ない悩みが足りない | C. 地雷・恥モジュール |
| B2Bや組織が絡むジャンル | D. B2B対応モジュール |

### A. 「抽象→具体」矯正モジュール

悩みがフワッとしがちな時の矯正。Phase Bの出力に対して追記する。

```text
抽象語が出たら、必ず具体化して言い直すこと。
具体化テンプレ：
「（いつ/どこで）〜している時に、（何が起きて）、（何に困る/何が怖い）」
例：
×「将来が不安」
○「このまま続けても成果が出る気がせず、時間だけ失って終わりそうで怖い」
```

### B. 「顧客の頭の中の独り言」増量モジュール

"本人語"の解像度を上げる。

```text
悩みはできるだけ「独り言」っぽく書く。
語尾例：
「…だったらどうしよう」「自分だけ置いてかれてる気がする」「また失敗しそう」「結局どれが正解なの？」
```

### C. 「地雷・恥・言いにくい悩み」掘り起こしモジュール

表に出にくい悩みを拾う。

```text
次に「人に言いにくい悩み」も30個追加して。
例：恥、劣等感、嫉妬、見栄、依存、過去の失敗、家族に言えない、周りにバレたくない、など。
```

### D. 「B2B対応」モジュール

副業・経営・営業コンサル等で、意思決定者と実務者で悩みが割れるのを拾う。

```text
もし対象がB2B/BtoC混在なら、悩みを以下の立場別にも出す：
- 本人（実行する人）の悩み
- 意思決定者（支払う/許可する人）の悩み
- 周囲（上司/家族/チーム/顧客）の反応による悩み
```

### 追い打ちプロンプト集（症状別）

```
■ 悩みが抽象的になった場合
  「全部の悩みに"いつ・どこで・何が起きて困る"を1文で付けて」

■ 似た悩みばかり出る場合
  「"停滞""失敗後""再挑戦"のステージだけで30個追加して」

■ 悩みが綺麗すぎて刺さらない場合
  「敬語禁止。独り言の口調で言い直して」

■ 出力が多すぎて使えない場合
  → Phase Dへ進む
```

## Phase D: 整形・クラスター化（飛ばし禁止）

### このPhaseの目的

大量の悩みを「使える形」に整理する。棚卸ししただけでは使えない。**必ずクラスター化する**。

### 整形プロンプト（原文のまま実行）

Phase B〜Cで出力された悩みリストに対して、以下を実行する。

```text
上で出した悩みリストを整理して、重複を統合しつつ「悩みクラスター」を12個作ってください。
各クラスターごとに以下を出す：

- クラスター名（短く）
- 代表的な本人語（3つ）
- 典型シーン（1つ）
- 背景要因（評価/制約/スキル/環境のどれが強いか）
- その悩みが解決すると得られる状態（ベネフィットを1文）
- 検索で使われそうなキーワード（5つ）
```

### クラスター化するとなぜ嬉しいか

```
- クラスター = そのまま「商品コンセプト」「講座の章立て」「発信カテゴリ」になる
- 代表的本人語 = LPや広告の見出しに使える
- 検索キーワード = SEOやYouTubeのネタになる
- ベネフィット = コンセプト設計の材料になる
```

### 追加で「刺さる悩みTop20」の抽出

クラスター化と並行して、特に強い悩みも保持する。

```
12クラスターに加え、「特に刺さりが強い悩み Top20」も残しておく。
Top20はコンテンツの優先ネタになる。
```

## Phase E: Deep Research版で根拠補強（任意）

### このPhaseの位置づけ

Phase B〜Dはブレスト（AI推定）ベース。
Phase Eは**実際の公開情報から「本人語の引用 + 出典」**を収集して補強する。

初心者はPhase Dまでで十分運用できる。
品質をさらに上げたい場合や、クライアントに根拠を示したい場合にPhase Eを実行する。

### おすすめ運用（最短で強い順番）

```
1. まず Phase B（汎用マスター）で「100〜150個」出す（発散）
2. 次に Phase D（整形プロンプト）で「12クラスター」にまとめる（使える形）
3. 必要なら Phase E（Deep Research版）で「本人語の引用」を集めて補強（根拠づけ）
```

### Deep Research用プロンプト（原文のまま実行）

```text
目的：{{ジャンル}}領域の見込み顧客が抱える悩みを、公開情報から「本人語」で収集し、頻出パターンを作る。

やること：
1) Q&Aサイト、掲示板、SNS、ブログ等から「悩みが書かれた文章」を収集
2) 悩みをパターン分類（統合してクラスター化）
3) 各クラスターに「代表的な本人語（短い引用）」と出典リンクを付ける
4) オンライン情報の偏り（愚痴が集まりやすい等）も注記する

出力：
- クラスター名（12〜20個）
- 代表的な本人語（引用2〜3個、短く）
- 典型シーン
- 背景要因の仮説（仮説は仮説と明記）
- よくある誤解/やりがちな失敗
- 出典リンク
```

### Deep Researchでテンプレ入力を自動埋めする方法

Phase Aの入力が不十分な場合、Deep Researchで逆算することもできる。

```text
目的：{{ジャンル}}領域の見込み顧客について、公開情報から
(1) よくある状態（ステージ）
(2) よく言われるゴール
(3) 代表的な制約
を抽出し、テンプレ入力欄を埋める。

やること：
- Q&A、掲示板、SNS、ブログなどから「悩み/目標/制約」が含まれる投稿を収集
- それらを頻出パターンにまとめる
- 可能なら本人語を短く引用し、出典リンクを添える

出力：
- ジャンル（定義）
- 顧客の状態：代表3タイプ（根拠つき）
- 顧客の望むゴール：代表5個（根拠つき）
- 顧客の制約：代表10個（根拠つき）
- 代表セグメント：2軸で提案
- 避けたい表現やNG：この領域で炎上しやすい言い方・誇大表現の例
```

## Phase F: ターゲットペルソナの設計

### このPhaseの目的

Phase B〜Eで生成した悩みデータを統合し、具体的な1人の人物像としてペルソナを設計する。

### ペルソナ生成テンプレート

```
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## よくある失敗と対処法

### 悩みが抽象的になる

```
原因：条件がふわっとしてる or ルールが弱い
対処：モジュールAを足す + 「シーン必須」にする
追い打ち例：「全部の悩みに"いつ・どこで・何が起きて困る"を1文で付けて」
```

### 似た悩みばかり出る

```
原因：ステージや種類の軸が埋まってない
対処：マスターの「ステージ × 種類」を明示して増やす
追い打ち例：「"停滞""失敗後""再挑戦"のステージだけで30個追加して」
```

### 悩みが綺麗すぎて刺さらない

```
原因：本人語になってない
対処：モジュールBを追加
追い打ち例：「敬語禁止。独り言の口調で言い直して」
```

### 出力が多すぎて使えない

```
原因：整理工程を飛ばしてる
対処：Phase Dの整形プロンプトでクラスター化する
追い打ち例：「重複を統合して、上位12クラスターに圧縮して」
```

## エラーハンドリング

### ジャンルが広すぎる場合

```
ジャンルが広いため、もう少し絞り込みが必要です。
例えば「マーケティング」の場合：
- SNSマーケティング
- コンテンツマーケティング
- 広告運用
- ブランディング
どの領域を中心にリサーチしますか？
```

### ジャンルがニッチすぎる場合

```
このジャンルは情報が限られるため、以下の方法でリサーチします：
1. 隣接ジャンルの悩みから類推する
2. 上位カテゴリの悩みを具体化する
3. ターゲット層の一般的な悩みと掛け合わせる

精度を上げるために、既に知っている悩みがあれば教えてください。
```

### キャラクター設計書がない場合

```
キャラクター設計書がなくても、ジャンル指定だけでリサーチできます。
ただし、キャラクター固有の接点設計（Phase F後半）は省略されます。
キャラクター設計後に「悩みを更新して」と指示すれば、接点設計を追加できます。
```

### 品質チェックリスト

```
生成した悩みデータベースの最終チェック：
□ 100個以上の悩みが出ているか
□ 悩みが「本人語」になっているか（抽象語で終わっていないか）
□ 偏りがないか（ステージ × 種類 × セグメントをカバーしているか）
□ 12クラスターに整理されているか
□ Top20が「確かに刺さる」と思えるか
□ ターゲットペルソナが現実的か（AI的な作り物になっていないか）
□ コンテンツ生成への接続情報が揃っているか
```
